1. Executive Summary
This document defines the complete scope of work, functional requirements, technical specifications,
and acceptance criteria for the development of an AI-powered Outbound Follow-Up Calling System.
The system will autonomously place outbound calls to clients retrieved from a managed database,
conduct natural English-language voice conversations using VAPI's AI voice agents, capture client
responses, handle rescheduling requests automatically, and log every interaction into structured
reports. A web-based dashboard will allow operators to manage clients, monitor call activity, and
download call summaries.
This is an MVP (Minimum Viable Product) but must be built to production standards — reliable,
observable, and ready for daily operational use at a volume of 200 to 500 calls per day.

2. Project Objectives
The system must achieve the following objectives:
1. Automate outbound follow-up calls to clients retrieved from a managed client database, with
zero manual dialing.
2. Deliver natural, human-like conversations in English using VAPI's AI voice agents, personalized
with each client's specific data and follow-up context.
3. Capture and structure client responses during every call, producing a clean summary of the
interaction.
4. Handle rescheduling requests automatically — if a client asks to be called at a later time, the
system must book that slot and place the call automatically at the requested time.
5. Provide a complete operator dashboard (UI) for client management, call scheduling, live
monitoring, and reporting.
6. Log every call outcome into a structured Excel/Google Sheets format for downstream review
and reporting.
7. Operate reliably at a daily volume of 200–500 calls with observable uptime and error handling.

4.2 AI Outbound Calling Engine
The core engine that places outbound calls via VAPI and runs the AI conversation.

4.2.1 Calling Engine Requirements
•

Integrate with VAPI using their official API/SDK.

•

Maintain a persistent call queue with status tracking per client.

•

A background scheduler must poll the queue and trigger calls at their scheduled time (respecting
the client's timezone).

•

Support concurrent outbound calls — target up to 10 simultaneous calls at peak hours.

•

Retry logic for failed calls: on no-answer/busy/voicemail, retry up to 3 times with configurable
delays (e.g., +30 min, +2 hr, next day same time).

•

Detect voicemail/answering-machine and handle it per configuration (leave message or hang
up).

•

Capture call duration, disposition, and recording URL for every attempt.

•

Respect calling-window restrictions (e.g., only call between 9am–8pm client local time) —
configurable per deployment.

4.2.2 VAPI Integration Requirements
•

Configure a VAPI Assistant for the follow-up use case with a system prompt that includes clientspecific context at runtime (dynamic variables).

•

Select an appropriate English voice model (e.g., ElevenLabs, PlayHT, or VAPI-native voice) —
finalize with the client during kickoff.

•

Handle VAPI webhooks: call-started, call-ended, message events, function-calls, and transcript
events.

•

Use VAPI's function-calling capability so the AI can trigger actions mid-call (e.g., bookreschedule, mark-not-interested, confirm-details).

•

Store all VAPI call IDs against our internal call records for auditability.

4.3 Conversation Flow Design (Follow-Up Use Case)
The AI agent's conversation must follow a consistent, natural flow for follow-up calls. The developer is
responsible for crafting the initial VAPI system prompt; the flow below is the required minimum
behavior.
4.3.1 Required Conversation Stages
8. Greeting & Identity Verification — The AI introduces itself by the configured business name and
confirms it is speaking to the correct client by name.
9. Purpose Statement — The AI clearly states the reason for the follow-up call, referencing the
follow_up_context field from the client record.
10. Permission to Proceed — The AI asks if it's a good time to talk. If no, it offers to reschedule (see
4.4).
11. Information Gathering — The AI asks the follow-up questions as per the conversation script,
capturing structured answers.
12. Summary & Confirmation — The AI repeats key answers back to the client for confirmation.
13. Reschedule or Next Action — If needed, the AI books a reschedule or confirms the next step.

14. Polite Closure — The AI thanks the client and ends the call.
4.3.2 Guardrails
•

The AI must never claim to be a human if asked directly.

•

The AI must stay on-topic — no open-ended chit-chat beyond the follow-up purpose.

•

The AI must handle interruptions gracefully (client talking over it).

•

Maximum call length — hard cutoff at 10 minutes per call.

•

If the client requests a human, the AI must log this and end the call politely.

4.4 Automatic Rescheduling Module
A critical feature: if a client requests to be called at a different time, the system must handle it entirely
without operator intervention.
4.4.1 Reschedule Flow
15. The AI detects a reschedule request (via function-calling or intent detection).
16. The AI asks for the preferred date and time.
17. The AI confirms the new slot back to the client.
18. The backend parses the date/time (with timezone), validates it, and persists the updated
scheduled_call_time.
19. The client's status is set to 'rescheduled'.
20. The scheduler will pick up the client again at the new time and call automatically.
4.4.2 Reschedule Rules
•

Maximum 3 reschedules per client (configurable). After that, the client is marked as 'manualfollow-up-required' for operator review.

•

Rescheduled times must fall within the configured calling window. If outside, the AI should offer
the next nearest valid slot.

•

All reschedule events must be logged with original time, new time, and reason.

4.5 Conversation Data Capture & Summarization
Every call must produce a structured record of what happened.
4.5.1 Captured Per Call
•

Full conversation transcript (from VAPI).

•

AI-generated summary of the call (3–5 sentences).

•

Structured answers to the follow-up questions (JSON/key-value).

•

Call disposition: completed, no-answer, voicemail, refused, rescheduled, failed.

•

Call duration, start/end timestamps.

•

Recording URL (if recording is enabled in VAPI).

•

Sentiment tag (positive / neutral / negative) — derived from the transcript.

6.2 High-Level System Architecture
The system must be built as the following logical components:
21. Web Dashboard (Frontend) — talks only to the Backend API.
22. Backend API — serves the dashboard, authenticates users, manages client records, exposes
reporting endpoints, and receives VAPI webhooks.
23. Scheduler Worker — a background job that continuously polls the call queue and triggers calls
whose scheduled_call_time has arrived (respecting timezone and calling window).
24. Call Orchestration Worker — initiates outbound calls via VAPI, listens for VAPI webhooks,
updates call state, triggers retries.
25. Post-Call Processor — on call-ended, fetches transcript, generates summary via LLM, extracts
structured answers, writes to DB and exports to Sheets/Excel.
26. Database — PostgreSQL with tables for clients, calls, call_events, reschedules, users, settings.
27. Message Queue (Redis) — decouples the scheduler from the orchestration worker and supports
retries and concurrency.

6.5 Capacity, Performance & Concurrency
•

Daily volume: 200–500 calls.

•

Peak concurrency: design for 10 simultaneous outbound calls. The calling engine must not
serialize calls.

•

Average call duration assumption: 2–3 minutes. Budget: 500 calls × 3 min ≈ 25 call-hours/day —
easily handled by VAPI + Twilio at modest concurrency.

•

Dashboard response time: < 500 ms p95 for list endpoints.

•

Webhook processing: VAPI webhook handler must respond < 200 ms and push heavy work to
the queue.

•

Scheduler tick: every 30 seconds (configurable).

7. Non-Functional Requirements
7.1 Security
•

All secrets (VAPI API key, Twilio credentials, database URL, OpenAI key) must be stored in
environment variables or a secrets manager — never in code.

•

The .env file must never be committed to Git. A .env.example must be provided instead.

•

HTTPS everywhere — no plain HTTP in production.

•

Passwords stored using bcrypt/argon2, never plaintext.

•

VAPI webhooks must be signature-verified.

•

Input validation on every API endpoint (zod, pydantic, or equivalent).

•

Rate limiting on auth and public endpoints.

•

Basic audit log: who created/edited/deleted clients and when.

7.2 Reliability
•

The calling engine must survive process restarts without dropping queued calls.

•

Failed VAPI calls must be retried per the retry policy.

•

Database backups: daily automated backups (managed DB providers handle this — confirm
enabled).

•

Graceful degradation: if the LLM summarization service is down, the call is still logged with raw
transcript; summarization retries later.

7.3 Observability
•

Structured JSON logging for all backend events.

•

Error tracking via Sentry or equivalent.

•

An uptime endpoint (/health) monitored by an external service.

•

Key metrics visible in the dashboard: calls initiated, calls completed, average duration, failure
rate.

7.4 Data Retention & Privacy
•

Call recordings and transcripts retained for a configurable period (default: 90 days).

•

Operator must be able to delete a client's data on request (GDPR-style delete).

•

Only authenticated operators may view client data or call recordings.

7.5 Maintainability
•

Clean, commented, readable code.

•

Consistent code style enforced by a linter + formatter (ESLint+Prettier, Ruff, etc.).

•

README in the repo with local setup steps runnable in < 15 minutes.

•

Environment-based configuration (dev / staging / production).
